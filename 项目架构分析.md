# llfcchat 项目架构分析（基于当前代码）

## 1. 项目整体架构总览

当前仓库是一个典型的“客户端 + 网关服务 + 验证码微服务”的三段式架构：

- `violetchat`：Qt 客户端（UI + HTTP 请求）
- `GateServer`：C++ 网关服务（HTTP 接入、业务路由、调用 Redis/MySQL/gRPC）
- `VarifyServer`：Node.js gRPC 服务（生成验证码、发邮件、写 Redis）

从职责上看：

1. 客户端只负责收集用户输入和展示结果。  
2. GateServer 承担“统一业务入口”，负责参数校验、验证码校验、注册入库。  
3. VarifyServer 专注验证码发放，不直接碰用户注册逻辑。

这已经具备了“基础微服务拆分”的雏形，方向是正确的。

---

## 2. 目录与模块职责

## 2.1 `violetchat`（Qt 客户端）

关键文件：

- `violetchat/registerdialog.cpp`：注册界面逻辑，触发“获取验证码”和“注册”两个 HTTP 请求。
- `violetchat/httpmgr.cpp`：统一 HTTP 发送器，异步请求结束后通过信号槽分发。
- `violetchat/global.h` / `violetchat/config.ini`：请求 ID、模块 ID、网关地址等配置。

职责定位：

- UI 交互层
- 请求发起层
- 响应分发层（按模块区分）

当前状态：客户端结构比较清晰，`HttpMgr` 做了请求统一收口，后续可以继续扩展登录、好友、消息模块。

## 2.2 `GateServer`（C++ 网关核心）

关键文件：

- 启动入口：`GateServer/src/test.cpp`
- 连接接入：`GateServer/src/CServer.cpp`
- 请求连接处理：`GateServer/src/HttpConnection.cpp`
- 路由/业务注册：`GateServer/src/LogicSystem.cpp`
- Redis 管理：`GateServer/src/RedisMgr.cpp`
- MySQL 管理：`GateServer/src/MysqlMgr.cpp`、`GateServer/src/MysqlDao.cpp`
- gRPC 客户端：`GateServer/src/VarifyGrpcClient.cpp`
- 配置中心：`GateServer/src/ConfigMgr.cpp`、`GateServer/src/config.ini`

职责定位：

- HTTP 统一接入
- 路由分发（URL -> Handler）
- 核心业务编排（验证码校验、账号注册）
- 外部资源访问统一封装（Redis / MySQL / gRPC）

## 2.3 `VarifyServer`（Node 验证码服务）

关键文件：

- 服务入口：`VarifyServer/server.js`
- 邮件发送：`VarifyServer/email.js`
- Redis 操作：`VarifyServer/redis.js`
- 配置：`VarifyServer/config.js`、`VarifyServer/config.json`

职责定位：

- 提供 `GetVarifyCode` gRPC 接口
- 从 Redis 读/写验证码
- 发送邮箱验证码

这个服务相当于“验证码中心”，和 GateServer 的业务边界目前是合理的。

## 3. GateServer 内部分层（重点）

从你现在的代码看，GateServer 可以拆成 5 层：

1. 接入层（Network Layer）  
2. 路由层（Routing Layer）  
3. 业务层（Service/UseCase Layer）  
4. 数据访问层（Repository/DAO Layer）  
5. 基础设施层（Infrastructure Layer）

## 3.1 接入层

文件：`CServer` + `HttpConnection`

- `CServer::Start()`：异步 accept 新连接。
- 每个连接交给 `HttpConnection`，由其负责读请求、写响应、超时处理。
- `HttpConnection::HandleReq()` 根据 HTTP method 分流到 `LogicSystem`。

特点：

- 基于 Boost.Asio + Beast，属于 Reactor 异步模型。
- `AsioIOServicePool` 让连接分散到多个 `io_context`，有并发扩展基础。

## 3.2 路由层

文件：`LogicSystem`

- `RegGet/RegPost` 注册 URL 对应处理函数。
- `HandleGet/HandlePost` 查表执行。

当前已注册路由：

- `POST /get_varifycode`
- `POST /user_register`
- `GET /get_test`

特点：

- 路由表驱动，后续加接口成本低。
- 但所有业务写在构造函数里，规模变大后会臃肿。

## 3.3 业务层

核心逻辑在 `LogicSystem.cpp` 的 lambda 中：

### A. 获取验证码 `/get_varifycode`

1. 解析 JSON。
2. 读取 `email` 字段。
3. 调用 `VerifyGrpcClient::GetVarifyCode(email)`。
4. 返回 `{error, email}`。

### B. 注册 `/user_register`

1. 解析 JSON（`user/email/passwd/confirm/varifycode`）。
2. 校验 `passwd == confirm`。
3. 去 Redis 读取 `code_<email>`。
4. 比对验证码。
5. 调用 `MysqlMgr::RegUser` 入库。
6. 返回注册结果。

这是标准的“编排式业务流程”，思路很清楚。

## 3.4 数据访问层

### Redis

- `RedisMgr` + `RedisConPool`（hiredis 连接池）
- 提供 `Get/Set/HSet/Del...` 统一接口

### MySQL

- `MysqlMgr`：轻量门面
- `MysqlDao`：实际执行 SQL（当前用存储过程 `reg_user`）
- `MysqlPool`：MySQL 连接池 + 保活/重连线程

当前你说“已经配置好了 mysql”，从代码层面看，你已经不仅是“连通”，而是走到了：

- 连接池
- DAO 封装
- 业务接入

这已经是中期阶段，不算初级 demo 了。

## 3.5 基础设施层

- `ConfigMgr`：读取 INI 配置，供各模块统一访问。
- `Singleton.h`：单例模板，服务模块复用。
- `const.h`：错误码、通用 include、工具定义。

## 4. 关键调用链（时序视角）

## 4.1 获取验证码链路

1. 客户端 `RegisterDialog` 点击“获取验证码”  
2. `HttpMgr` 发 `POST /get_varifycode`  
3. Gate `LogicSystem` 调用 gRPC 客户端  
4. Gate -> VarifyServer: `GetVarifyCode(email)`  
5. VarifyServer:
   - 查 Redis 是否已有验证码
   - 无则生成 4 位码并写入 Redis（过期时间）
   - 调用 `nodemailer` 发邮件
6. 返回 GateServer，再回客户端

## 4.2 用户注册链路

1. 客户端填写资料后发 `POST /user_register`。
2. GateServer 参数校验。
3. GateServer 从 Redis 读取 `code_<email>` 并校验。
4. 校验通过后调用 `MysqlDao::RegUser`。
5. `RegUser` 通过存储过程写入数据库并返回 uid。
6. GateServer 返回注册成功 JSON。

## 6. 当前架构里需要优先关注的问题

以下是结合现有代码的高优先级问题：

1. `LogicSystem` 业务与路由强耦合  
现状：所有路由和业务 lambda 写在构造函数。  
风险：接口一多就会难维护、难测试。

2. 返回 JSON 泄露敏感字段  
现状：注册成功时把 `passwd/confirm/varifycode` 回传给客户端。  
风险：安全问题，不应返回。

3. Redis 过期时间与文案不一致  
现状：`VarifyServer` 设置 600 秒，但邮件文案写“三分钟内”。  
风险：用户体验和排障混乱。

4. `RedisConPool` 的 `b_stop_` 未显式初始化  
现状：构造函数初始化列表没有 `b_stop_(false)`。  
风险：并发等待条件可能出现未定义行为。

5. 数据层返回值语义不统一  
现状：`RegUser` 里既有 `0/-1/uid`，又有 `return false`（被当 int 用）。  
风险：调用方判断逻辑不稳，易埋 bug。

6. 错误码体系跨服务未统一  
现状：Qt、Gate、Varify 都有各自错误码定义。  
风险：前后端错误语义漂移，后续联调成本高。

7. `ConfigMgr::operator[]` 会在 key 不存在时隐式插入  
风险：配置拼写错误时不会立刻失败，故障隐蔽。

---

## 7. 推荐重构路线（按优先级）

## 第一阶段（先做，低成本高收益）

1. 注册响应去敏感字段（不回传密码、验证码）。
2. 统一验证码有效期配置（文案与 Redis TTL 一致）。
3. 修复 `RedisConPool` 初始化与资源关闭细节。
4. 明确 `RegUser` 的返回契约（如：`>0 成功，<=0 失败`，并固定错误码映射）。

## 第二阶段（结构优化）

1. 把 `LogicSystem` 的 lambda 拆到独立 handler 文件：
   - `UserHandler`（注册、登录）
   - `VerifyHandler`（验证码）
2. 引入 `Service` 层：
   - `UserService`
   - `VerifyService`
3. `LogicSystem` 只负责路由，不直接写业务细节。

## 第三阶段（可扩展能力）

1. 抽象统一响应体：`{code, msg, data}`。
2. 增加日志分级（info/warn/error）和请求追踪 id。
3. 为 MySQL/Redis/gRPC 增加超时与熔断策略。
4. 增加单元测试和接口测试（至少覆盖注册主流程）。

---

## 8. 给你当前阶段的建议

你现在最需要掌握的是这三件事：

1. “接入层不写业务，业务层不写数据库细节”的分层边界。  
2. “业务编排”与“资源访问封装”的职责分离。  
3. “统一错误码 + 统一响应结构”的契约意识。

只要按上面的第一阶段先做完，你这个项目的工程质量会立刻上一个台阶，而且不会影响你后续继续做登录、好友、聊天消息等功能。

## 9. 你可以把当前架构记成一句话

`Qt 客户端发 HTTP -> GateServer 做路由与业务编排 -> 通过 Redis 校验验证码 + 通过 MySQL 存储用户 + 通过 gRPC 调验证码服务。`

这句就是你后续和别人讲项目时最清晰的“电梯陈述”。
